# popMind 本地小模型接入实施文档

## 1. 目标

本文档的目标不是给项目新增一整套新的本地模型 Provider。

本文档要解决的是一件更直接的事：

1. 让当前 `popMind` 尽快接入本地小模型
2. 优先支持简单文本能力
3. 尽量少改代码，先把能力跑通
4. 后续再补“本地模型专属体验”

本期建议的落地范围：

1. 划词后的本地解释 / 简单问答
2. 轻量翻译
3. 总结、改写、简短代码解释

本期明确不作为交付目标的内容：

1. 图片理解稳定支持
2. 音频输入
3. function calling / agent 工具调用
4. 本地模型专属设置页
5. 自动识别所有本地模型能力

## 2. 结论

当前项目已经具备接入本地模型的基础能力。

最省事的方案不是新增 `GemmaProvider`，而是把本地模型通过 OpenAI 兼容接口接入现有 `openai` 配置入口。

推荐优先级：

1. 首选 `LM Studio`
2. 次选其他能提供 OpenAI 兼容接口的本地服务

原因很简单：

1. 当前设置页已经支持填写 `provider`、`baseURL`、`model`
2. 当前 `provider-factory` 已经支持 `OpenAI compatible` 的 `baseURL`
3. 当前测试链路已经可以直接验证模型是否可调用
4. 当前解释链路已经统一走 AI 模型入口

换句话说，当前项目要做“本地简单问答”，一期可以不改主流程代码，先通过配置落地。

## 3. 当前代码基线

### 3.1 已有能力

当前项目已经有下面这几段关键基础：

1. 设置页支持选择 `OpenAI`、填写 `API Key`、`baseURL`、`model`
2. `OpenAI` 配置支持自定义 `baseURL`
3. 非 OpenAI 官方域名时，会按 OpenAI 兼容方式请求
4. 设置页里已经有 `Test` 按钮验证可用性
5. 划词解释已经统一走模型工厂

关键位置：

1. `app/components/settings/SettingsPage.tsx`
2. `lib/ai-service/provider-factory.ts`
3. `lib/ai-service/test-service.ts`
4. `lib/explain/runner.ts`

### 3.2 当前限制

虽然文本能力今天就能接，但当前实现有几个明确限制：

1. 本地模型没有独立入口，用户体验上是借用 `OpenAI` 选项
2. 走 `openai` 入口时，图片输入能力按 OpenAI 模型名规则识别，Gemma 4 不会被正确识别为支持视觉
3. 上下文上限按 provider 固定，`openai` 入口当前按 `128K` 处理
4. function calling 目前没有完整接通，所以 Gemma 原生工具调用能力暂时用不上

因此，本期应该把目标明确限定为“本地文本能力先跑通”。

## 4. 推荐方案

### 4.1 技术路线

本期推荐路线：

1. 在本机安装并启动 `LM Studio`
2. 下载并加载 `Gemma 4 E2B` 或 `Gemma 4 E4B`
3. 打开 LM Studio 的本地 OpenAI 兼容服务
4. 在 `popMind` 设置页选择 `OpenAI`
5. 将 `baseURL` 指向本地服务地址
6. 将 `model` 填为当前加载的模型名
7. 用当前项目已有的测试按钮验证连通性

### 4.2 为什么不先新写 Provider

本期不建议先做单独的 `Gemma / Ollama / LM Studio` Provider，原因是：

1. 成本更高
2. 用户价值提升有限
3. 当前项目已经能用最小改动跑通主流程
4. 先验证“本地简单问答”是否真的满足使用场景更重要

正确节奏应该是：

1. 先通过兼容接口把功能跑通
2. 再决定要不要产品化成独立入口

## 5. 模型选择建议

如果目标是“本地简单工作”，建议优先使用小模型：

1. `Gemma 4 E2B`
2. `Gemma 4 E4B`

建议：

1. `16GB` 机器优先试 `E2B`
2. `24GB` 或 `32GB` Apple Silicon 优先试 `E4B`
3. `26B / 31B` 不作为一期默认方案

原因：

1. 这次目标是简单文本能力，不是极限效果
2. 小模型启动更快，资源压力更低
3. 更适合先验证划词解释、翻译、总结这类轻任务

## 6. 实施步骤

### 6.1 阶段 A：零代码接入

这个阶段的目标是“不改主流程代码，直接让本地小模型可用”。

### 步骤 1：准备本地模型服务

开发者需要在本机完成：

1. 安装 `LM Studio`
2. 下载 `Gemma 4 E2B` 或 `Gemma 4 E4B`
3. 在 LM Studio 中启动本地服务
4. 确认服务地址可用

常见本地地址示例：

```text
http://127.0.0.1:1234/v1
```

实际以本机 LM Studio 显示地址为准。

### 步骤 2：在项目中配置 AI 服务

打开 `popMind` 设置页，按下面方式填写：

1. `Provider` 选择 `OpenAI`
2. `API Key` 填任意非空值，例如 `local`
3. `Base URL` 填本地 OpenAI 兼容地址
4. `Model` 填 LM Studio 当前暴露的模型名

示例：

```text
Provider: OpenAI
API Key: local
Base URL: http://127.0.0.1:1234/v1
Model: gemma-4-e4b-it
```

注意：

1. `API Key` 在当前项目里主要用于通过非空校验
2. `Model` 必须和本地服务真实暴露的模型标识一致
3. 如果服务端模型名不同，必须按服务端实际值填写

### 步骤 3：点击测试

使用设置页里的 `Test` 按钮验证。

当前测试逻辑会发起一条最小请求，成功返回即表示：

1. 项目可以连接本地服务
2. 模型名配置正确
3. 当前 Provider 链路可用

### 步骤 4：验证业务功能

建议按下面顺序验证：

1. 划词解释
2. 多轮追问
3. 简单翻译
4. 总结 / 改写

建议优先验证的输入：

1. 一段英文技术文档
2. 一段中文说明文字
3. 一小段代码
4. 一段长度中等的多句文本

### 6.2 阶段 B：文档化和产品说明

这个阶段不强制改代码，但建议补齐对外说明。

建议补充内容：

1. 设置页文案里说明 `OpenAI` 也支持本地兼容服务
2. 在用户文档中增加 LM Studio 配置示例
3. 说明一期仅保证文本能力

### 6.3 阶段 C：后续产品化改动

如果阶段 A 跑通并确认要长期支持本地模型，建议补下面几项小改动。

### 改动 1：新增本地模型独立入口

目标：

1. 设置页里新增 `Local / LM Studio / Ollama` 之类的显式入口
2. 不再让用户误以为只能连接 OpenAI 官方

建议做法：

1. 新增本地模型 Provider 类型，底层仍复用 OpenAI 兼容客户端
2. 在 UI 上把“本地服务地址”和“模型名”文案写清楚

### 改动 2：补 Gemma 能力识别

目标：

1. 正确识别 Gemma 视觉模型
2. 避免误判为纯文本模型

建议改动点：

1. `lib/ai-service/provider-factory.ts`
2. 调整 `supportsImageInput(...)` 的模型名判断规则

### 改动 3：支持可配置上下文上限

目标：

1. 不再把所有本地兼容模型都固定按 `128K` 处理
2. 允许为不同模型设置更合适的上下文上限

建议做法：

1. 在设置中增加可选的 `contextLimitOverride`
2. 或增加 Gemma 模型预设

### 改动 4：补更明确的错误提示

目标：

1. 用户能区分是服务没启动
2. 还是地址错误
3. 还是模型名错误

建议改动点：

1. `lib/ai-service/test-service.ts`
2. 设置页测试结果提示文案

## 7. 能力范围说明

本期接入后，建议承诺的能力边界如下。

### 7.1 本期可用

1. 划词解释
2. 简单问答
3. 简单翻译
4. 总结、改写、释义
5. 简短代码解释

### 7.2 本期不承诺

1. 图片理解一定可用
2. OCR 一定可用
3. 超长上下文一定按 Gemma 理论上限跑满
4. function calling 一定可用
5. 所有本地服务实现都完全兼容

## 8. 验收标准

只要满足下面条件，就可以认为“一期本地小模型接入完成”：

1. 本机本地模型服务可启动
2. 设置页 `OpenAI + Base URL + Model` 可保存
3. `Test` 按钮通过
4. 划词解释能正常返回内容
5. 多轮追问能继续响应
6. 翻译链路能正常返回文本结果
7. 失败时用户能看到明确报错，而不是静默失败

## 9. 开发顺序建议

推荐按下面顺序推进：

1. 先完成阶段 A，验证零代码接入
2. 确认本地小模型效果可接受
3. 再决定是否做阶段 C 的产品化改动

不要一开始就做大改。

先把“能不能稳定完成简单文本工作”验证清楚，再决定是否扩展成本地模型正式能力。

## 10. 后续任务建议

如果要继续推进下一轮开发，建议任务拆分如下：

1. 设置页增加“本地模型”专属说明
2. provider-factory 增加 Gemma 模型能力识别
3. 增加上下文上限覆盖配置
4. 增加 LM Studio / Ollama 预设地址与示例模型名
5. 增加一份面向用户的使用说明

## 11. 附录：推荐默认配置

推荐先给开发者一个能最快试通的默认组合：

```text
本地服务: LM Studio
模型: Gemma 4 E2B / Gemma 4 E4B
Provider: OpenAI
API Key: local
Base URL: http://127.0.0.1:1234/v1
用途: 划词解释、简单问答、简单翻译、摘要改写
```

这个组合的优点是：

1. 改动最少
2. 最容易验证
3. 和当前项目结构最匹配
4. 足够支撑“本地简单工作”的一期目标
